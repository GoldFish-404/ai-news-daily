# 开发经验总结：AI 资讯聚合站项目

> 从本项目对话中提炼的经验和踩坑记录。每个项目有自己独立的 dev-lessons.md，后续可跨项目汇总成规范性文档。

---

## 1. 项目启动阶段

### 先沟通方案再写代码
- 本次用了计划模式（EnterPlanMode），在动手前明确技术栈、数据流、UI 布局
- 用户对技术栈不熟悉时，直接给出推荐方案而非罗列选项让用户选
- **规则**：大型任务先出计划等审批，简单修改直接做

### 一次性确认所有环境依赖
- 项目需要的外部账号/Key 提前列出：GitHub 账号、Supabase 项目、Vercel 账号
- 不要等代码写到一半才发现缺环境
- **规则**：启动时列清单问用户"你有哪些，还缺哪些"

---

## 2. 技术选型经验

### 针对中国网络环境的特殊处理
- **Google Fonts**：国内加载失败，直接使用系统字体 `font-sans`
- **GitHub API**：本地 curl 可能 SSL 错误，Node.js 需加 `--use-system-ca`
- **Vercel 默认域名**：`.vercel.app` 在国内被墙，需提前告知用户绑域名
- **规则**：面向国内用户的项目，默认假设网络受限

### Serverless 超时限制
- Vercel 免费版函数超时 10 秒，爬虫需控制并发请求数
- HN 从 100 条降至 50 条就是为了适配这个限制
- **规则**：涉及外部网络请求的 Cron/API，预估耗时并适配平台限制

---

## 3. 用户反馈模式

### 用户偏好
- **不要频繁让用户操作**（"不要什么都让我来"）——能用 Token/CLI 搞定的就直接搞定
- **可以给账号，不给密码**——用户愿意提供 Token、Key 等，但不希望自己操作
- **英文内容可接受，但 UI 必须中文**——信息源的原始内容保持英文，界面和辅助文字要中文
- **简单优先**——UI 极简、一张数据库表、不做分页、不做复杂功能

### 沟通节奏
- 用户说"我不了解"时不要继续追问细节，直接给推荐方案
- 关键决策点确认（如：推送方式、部署平台），细节自行决定
- 每次改动后告知用户在做什么，但不要冗长解释

## 4. 踩坑记录

### Supabase RLS 问题
- **现象**：数据写入 API 返回成功但 saved: 0
- **排查**：先写测试脚本直接调 Supabase，发现 RLS 拦截
- **解决**：service_role key 绕过 RLS，anon key 只用于前端读取
- **教训**：Supabase 建表后默认开启 RLS，第一时间处理权限问题

### SSR 构建失败排查
- **现象**：Vercel 构建失败，本地构建正常
- **根因**：环境变量未在 Vercel 配置，`page.tsx` 在构建阶段调 Supabase 时找不到 KEY
- **解决**：`vercel env add` 批量添加环境变量
- **教训**：部署前把 `.env.local` 的所有变量同步到平台

### 敏感信息管理
- 测试脚本中硬编码的 Key 用完及时删除
- Git remote URL 中嵌入的 Token 用完移除
- `.env.local` 确保在 `.gitignore` 中
- **教训**：每次"修 bug 的临时脚本"用完后立即清理

## 5. 高效开发流程（建议）

1. **计划阶段**：出方案 → 等审批（不等就开始写 = 返工）
2. **环境阶段**：一次性列出所有需要的账号/Key → 让用户一次性提供
3. **编码阶段**：按模块自底向上（类型定义 → 数据源 → 存储 → API → UI）
4. **测试阶段**：本地跑通全链路后再部署
5. **部署阶段**：先配环境变量再部署，关注平台特有的限制（超时、网络）
6. **收尾阶段**：清理临时文件、写项目文档、总结踩坑

## 6. 版本管理经验

### Git 初始化
- 用户之前未用过 Git，需帮用户配好 `user.name` 和 `user.email`
- 新项目启动时第一步就是 `git init` + 配 `.gitignore`
- **规则**：项目目录创建后立即初始化 Git，不等代码写完

### GitHub 仓库创建
- 用户提供 Personal Access Token（Classic，勾选 repo 权限）即可代操作
- 国内访问 GitHub API 可能 SSL 失败，用 `node --use-system-ca` 解决
- 创建仓库 → 设 remote → push，全流程命令行完成
- **规则**：用户提供 Token 即可全权处理 GitHub 操作，不要让用户去网页手动创建

### 敏感信息处理
- Token 嵌在 remote URL 用完即移除：`git remote set-url origin https://github.com/...`（去掉 user:token@）
- `.env.local` 必须进 `.gitignore`，提交前用 `git status` 确认
- 临时测试脚本（含硬编码 Key）用完立即删除
- **规则**：每次 push 前检查是否有敏感信息残留

### 文档版本管理策略
- 每个项目独立维护 `docs/project-log.md` + `docs/dev-lessons.md`
- 文档是"我和用户沟通的成果"：存本地，不进 Git（除非用户明确说推）
- 当前在哪个项目工作，就维护哪个项目的文档
- 多个项目完成后可跨项目汇总成规范性文档

### 推送时机
- 每个独立修改完成后立即 commit + push（不要攒一批）
- commit message 用英文简述改动内容
- **规则**：一个逻辑改动 = 一个 commit = 一次 push

## 7. 文档维护机制

### 每个项目独立维护两份文档

```
<项目>/
  docs/
    project-log.md    ← 项目进展（方案、修改、状态、待办）
    dev-lessons.md    ← 本项目开发经验（踩坑、偏好、规律）
```

### 项目启动前
- 创建 `<项目>/docs/` 目录和两份文档的初始版本
- project-log.md 写入初始方案
- dev-lessons.md 写入模板框架
- 告知用户文档已就绪

### 对话进行中
- 每个重要改动（方案变更、Bug 修复、用户反馈）及时写入 project-log.md
- 用户说的"记住这个"、"以后都这样"类反馈，写入 dev-lessons.md
- 写入后简短告知用户

### 项目收尾时
- 补充完整 project-log.md 的当前状态和待办事项
- 从本次对话提炼新规律，更新 dev-lessons.md
- 告诉用户两份文档已更新

### 跨项目汇总（远期）
- 多个项目完成后，用户可要求从各项目的 dev-lessons.md 汇总出规范性文档
